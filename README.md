# SHL Assessment Recommendation Agent

A conversational agent that helps hiring managers find the right **SHL
Individual Test Solutions** for a role, through a back-and-forth chat
instead of manual keyword search. Built for the SHL AI Intern
take-home assignment.

## Project Overview

You describe a role ("I'm hiring a Java developer"), and the agent:

- Asks one clarifying question at a time if it doesn't have enough
  information yet.
- Retrieves the most relevant assessments from the SHL catalog using
  semantic search (embeddings + FAISS).
- Asks Google Gemini to turn those retrieved candidates into a natural
  reply and a shortlist - Gemini is never allowed to recommend
  anything outside what was actually retrieved.
- Supports refining the shortlist mid-conversation and comparing two
  assessments.
- Refuses off-topic questions and prompt-injection attempts.

## Architecture

```
User message
     |
     v
 guardrails.py  --(off-topic / injection)--> canned refusal
     |
     v
 retriever.py   -- semantic search over data/shl_catalog.json (FAISS)
     |
     v
 chatbot.py     -- builds a grounded prompt, calls Gemini,
     |              validates Gemini's picks against the real catalog
     v
 routes.py      -- FastAPI /chat endpoint, returns schema-valid JSON
```

This is a classic **RAG (Retrieval-Augmented Generation)** shape: we
retrieve first with an embedding model, and only ever show the LLM
assessments that actually exist in our data, so it can't hallucinate
a test name or a URL. Even so, `chatbot.py` double-checks Gemini's
output against the real catalog before it ever reaches the user - the
prompt is the first layer of defence, the code is the real guarantee.

## Installation

```bash
git clone <your-repo-url>
cd shl-assessment-agent
python -m venv venv
source venv/bin/activate      # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

## Environment Variables

Copy `.env.example` to `.env` and fill in your key:

```bash
cp .env.example .env
```

| Variable | Description | Default |
|---|---|---|
| `GEMINI_API_KEY` | Your Google Gemini API key (get one free at [aistudio.google.com](https://aistudio.google.com/app/apikey)) | *(required)* |
| `GEMINI_MODEL_NAME` | Gemini model to use | `gemini-2.5-flash` |
| `EMBEDDING_MODEL_NAME` | sentence-transformers model | `all-MiniLM-L6-v2` |
| `TOP_K` | How many catalog items to retrieve per query | `10` |
| `CATALOG_PATH` | Path to the catalog JSON file | `data/shl_catalog.json` |
| `MAX_TURNS` | Max conversation turns (matches SHL's 8-turn evaluator cap) | `8` |

## Running Locally

```bash
uvicorn app.main:app --reload
```

The API will be available at `http://127.0.0.1:8000`. Interactive docs
are auto-generated by FastAPI at `http://127.0.0.1:8000/docs`.

## Deployment on Render

1. Push this repo to GitHub.
2. In Render, create a **New Web Service** and connect the repo (or
   just run `render deploy` if you use the Render CLI - `render.yaml`
   is already set up as a Blueprint).
3. Render will auto-detect `render.yaml` and use:
   - Build command: `pip install -r requirements.txt`
   - Start command: `uvicorn app.main:app --host 0.0.0.0 --port $PORT`
4. Add `GEMINI_API_KEY` as an environment variable in the Render
   dashboard (it's marked `sync: false` in `render.yaml` so it isn't
   committed to the repo).
5. Once deployed, confirm both endpoints are reachable:
   - `GET https://<your-app>.onrender.com/health`
   - `POST https://<your-app>.onrender.com/chat`

Render's free tier spins down when idle, so the first `/health` call
after a period of inactivity can take up to ~1-2 minutes to wake up -
this matches the grace period the SHL evaluator allows.

## Refreshing the Catalog

`data/shl_catalog.json` currently ships with a hand-curated sample of
SHL Individual Test Solutions (see `approach_document.md` for why).
To pull the live catalog instead:

```bash
python scripts/scrape_catalog.py
```

This overwrites `data/shl_catalog.json` with freshly scraped data.
Restart the server afterwards so the FAISS index gets rebuilt from the
new data.

## Future Improvements

- Cache Gemini responses for identical conversation states to cut
  latency and cost on repeated evaluator runs.
- Swap the flat FAISS index for an HNSW index if the catalog grows
  much larger than a few hundred items.
- Add a small evaluation script that runs the public conversation
  traces automatically and reports Recall@K, instead of checking them
  by hand (see `tests/sample_conversations.md`).
- Persist conversation-level metrics (e.g. how often the agent asks
  more than one clarifying question) to catch prompt regressions early.
